# Dicionário

Cada tópico aqui listado facilmente resultaria em uma enorme dissertação, mas o intuito desse artigo não é esse. O meu objetivo aqui é o mesmo que o seu, aprender como lidar de forma correta com operações assíncronas de alta vazão em *API's*, tal como entender boas práticas de arquitetura e engenharia de software. Portanto, uma breve introdução já se faz suficiente, semelhante à uma breve busca na [Wikipedia](https://pt.wikipedia.org/wiki/Wikip%C3%A9dia:P%C3%A1gina_principal).

- [I/O bound](https://en.wikipedia.org/wiki/I/O_bound): operações que são limitadas pelo tempo de computagem

```
Legenda:
[###] = Trabalhando
[...] = Idle

Tempo ---------------------------------------------------------------------->

App / CPU  : [###] [###] [........................................] [###] [###]
             Req   Cons                   Idle                      Resp  Retorno

Banco / IO :             [########################################]
                                         Busca
```

- [CPU-bound](https://en.wikipedia.org/wiki/CPU-bound): operações que são limitadas puramente pela velocidade do processador, ou seja, a operação consome todo o poder computacional do CPU por um tempo indeterminado.

```
Tempo ---------------------------------------------------------------------->

App / CPU  : [###] [################################################] [###]
             Req   Cálculo de Matrizes (Pandas)                       Retorno

Disco / IO : (Vazio - Não há espera externa, os dados já estão na memória)
```

- [DI -- Dependency Injection](https://en.wikipedia.org/wiki/Dependency_injection): conceito de engenharia de software que visa diminuir o acoplamento de funções em dependências da aplicação, geralmente por meio de interfaces -- quem usa não sabe da implementação, apenas dos métodos que foram definidos nela -- quem programa apenas injeta um objeto que implementa essa interface dentro da função.

```python
# Sem DI
class Pinscher():
    def latir(self):
        return "AU"
        
def not_dependency_injection() -> str:
    """Código altamente acoplado na variável pinscher"""
    pinscher = Pinscher()
    return pinscher.latir()
    
if __name__ == "__main__":
    print(not_dependency_injection())
```

```python
# Com DI
from abc import ABC, abstractmethod

class Cachorro(ABC):
    @abstractmethod
    def latir(self):
        pass
        
class Pinscher(Cachorro):
    def latir(self):
        return "AU"

def dependency_injection(cachorro: Cachorro) -> str:
    """
    Código nada acoplado na variável pinscher
    
    A API sabe apenas como funciona a interface
    de forma abstrata, mas quem injeta é o usuário
    """
    return cachorro.latir()

if __name__ == "__main__":
    pinscher = Pinscher()
    print(dependency_injection(pinscher))

```

- [Async I/O](https://en.wikipedia.org/wiki/Asynchronous_I/O): Pode ser resumido mal e porcamente como um I/O que não necessariamente espera a finalização de um processo anterior para ser iniciado, ele declara quais processos já foram finalizados na execução, e assim os seus sucessores podem fazer proveito desses recursos, geralmente esses status são comunicados por meio do *polling*.

- [Polling](https://en.wikipedia.org/wiki/Polling_(computer_science)): Monitoramento do status de um recurso, para uma finalidade. O computador aguarda até o I/O ser finalizado -- podendo executar outras tarefas enquanto aguarda -- e monitora o estado do processo via *polling*.

# SQL Alchemy

Essa lib serve o propósito de ser uma interface para construção de [ORM's](https://en.wikipedia.org/wiki/Object%E2%80%93relational_mapping) em Python, e de forma ["pythônica"](https://www.reddit.com/r/learnprogramming/comments/1ps61gp/comment/nv724n0/?utm_source=share&utm_medium=web3x&utm_name=web3xcss&utm_term=1&utm_content=share_button). Partindo disso, podemos trazer a tona um debate muito recorrente na comunidade de programação: ORM's causam mais problemas do que consertam? Sendo bem direto ao ponto, o fato de uma ORM existir é para adicionar um nível a mais de abstração para desviar o foco do programador para coisas mais "importantes" naquele momento, existem diversos relatos envolvendo queries complexas usando ORM's que tendem a ter uma performance significativamente pior do que se a mesma fosse montada cuidadosamente pelo desenvolvedor -- levemos em considerção que esses casos são mais frequentes em aplicações que já escalaram, ou seja, existe um produto consolidado e funcionando. Dentro desse tópico, existe uma ótima palestra dada por um dos criadores da linguagem [Gleam](https://gleam.run/), onde ele disserta sobre uma biblioteca que tem o intuito de inverter a forma como lidamos com ORM's --> [Clique aqui para abrir a palestra](https://www.youtube.com/watch?v=8vrEtKc-TBU). Essa pergunta não pode ser respondida com um mero "sim" ou "não", esse é um tema complexo -- são trade-offs, assim como tudo na computação.
