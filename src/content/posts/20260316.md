## TL;DR
Nesse post introduzo e disseco vantagens de sistemas que usam mensagerias para gerenciamento de operações I/O de forma assíncrona. Também te explico como possivelmente a forma que você lida com suas ORM's pode estar causando um gargalo desnecessário dentro da sua aplicação.

# Glossário
Cada tópico aqui listado facilmente resultaria em uma enorme dissertação, mas o intuito desse artigo não é esse. O meu objetivo aqui é o mesmo que o seu, aprender como lidar de forma correta com operações assíncronas de alta vazão em *API's*, tal como entender boas práticas de arquitetura e engenharia de software. Portanto, uma breve introdução já se faz suficiente, semelhante à uma breve busca na [Wikipedia](https://pt.wikipedia.org/wiki/Wikip%C3%A9dia:P%C3%A1gina_principal).

- [I/O bound](https://en.wikipedia.org/wiki/I/O_bound): operações que são limitadas pelo tempo de espera

```
Legenda:
[###] = Trabalhando
[...] = Idle

Tempo ---------------------------------------------------------------------->

App / CPU  : [###] [###] [........................................] [###] [###]
             Req   Cons                   Idle                      Resp  Retorno

Banco / IO :             [########################################]
                                         Busca
```

- [CPU-bound](https://en.wikipedia.org/wiki/CPU-bound): operações que são limitadas puramente pela velocidade do processador, ou seja, a operação consome todo o poder computacional do CPU por um tempo indeterminado.

```
Tempo ---------------------------------------------------------------------->

App / CPU  : [###] [################################################] [###]
             Req   Cálculo de Matrizes (Pandas)                       Retorno

Disco / IO : (Vazio - Não há espera externa, os dados já estão na memória)
```

- [DI -- Dependency Injection](https://en.wikipedia.org/wiki/Dependency_injection): conceito de engenharia de software que visa diminuir o acoplamento de funções em dependências da aplicação, geralmente por meio de interfaces -- quem usa não sabe da implementação, apenas dos métodos que foram definidos nela -- quem programa apenas injeta um objeto que implementa essa interface dentro da função.

```python
# Sem DI
class Pinscher():
    def latir(self):
        return "AU"
        
def not_dependency_injection() -> str:
    """Código altamente acoplado na variável pinscher"""
    pinscher = Pinscher()
    return pinscher.latir()
    
if __name__ == "__main__":
    print(not_dependency_injection())
```

```python
# Com DI
from abc import ABC, abstractmethod

class Cachorro(ABC):
    @abstractmethod
    def latir(self):
        pass
        
class Pinscher(Cachorro):
    def latir(self):
        return "AU"

def dependency_injection(cachorro: Cachorro) -> str:
    """
    Código nada acoplado na variável pinscher
    
    A API sabe apenas como funciona a interface
    de forma abstrata, mas quem injeta é o usuário
    """
    return cachorro.latir()

if __name__ == "__main__":
    pinscher = Pinscher()
    print(dependency_injection(pinscher))

```

- [Async I/O](https://en.wikipedia.org/wiki/Asynchronous_I/O): Pode ser resumido mal e porcamente como um I/O que não necessariamente espera a finalização de um processo anterior para ser iniciado, ele declara quais processos já foram finalizados na execução, e assim os seus sucessores podem fazer proveito desses recursos, geralmente esses status são comunicados por meio do *polling*.

- [Polling](https://en.wikipedia.org/wiki/Polling_(computer_science)): Monitoramento do status de um recurso, para uma finalidade. O computador aguarda até o I/O ser finalizado -- podendo executar outras tarefas enquanto aguarda -- e monitora o estado do processo via *polling*.

# SQL Alchemy
O SQL Alchemy em seu cerne é apenas uma interface de programação a.k.a. "[ORM's](https://en.wikipedia.org/wiki/Object%E2%80%93relational_mapping)" em Python, e de forma ["pythônica"](https://www.reddit.com/r/learnprogramming/comments/1ps61gp/comment/nv724n0/?utm_source=share&utm_medium=web3x&utm_name=web3xcss&utm_term=1&utm_content=share_button). Dito isso, o fato de ORM's criarem uma camada a mais de abstração para o desenvolvedor comunica algo bem importante: o uso dessa ferramenta serve o de melhorar a experiência de desenvolvimento no curto prazo. Recentemente assisti uma ótima palestra dada por um dos criadores da linguagem [Gleam](https://gleam.run/) -> [Clique aqui para abrir a palestra](https://www.youtube.com/watch?v=8vrEtKc-TBU).

ORM's não abstraem os problemas de I/O que existem, muito pelo contrário, pioram dependendo da forma como são usadas. O ponto aqui é: podemos usar o SQL Alchemy de muitas maneiras, a mais comum envolve abrir uma session para cada insert de model no banco, geralmente configuramos isso através de uma model base que vai herdar para todas as outras -- temos métodos genéricos de CRUD que fazem o gerenciamento de sessões automaticamente, conhecemos esse padrão como "repository pattern".

## Problemas com ORM's
Tá, mas quais os problemas que podem ser desencadeados no uso de ORM's? Essas abstrações são ótimas para o DX, mas podem causar um gargalo absurdo em pontos críticos da aplicação que necessitam de alta vazão dos dados que chegam. Pense da seguinte forma: tenho uma API que faz o papel de receber, processar, estruturar e armazenar dados em um banco SQL, podemos identificar os possíveis pontos críticos apenas com essa descrição...

```
[ Requisições HTTP / Alta Concorrência ]
           |
           v
+-----------------------------------------------------------------------+
|                                API                                    |
|                                                                       |
|  +=================================================================+  |
|  ||                  ABERTURA DE SESSÃO (ORM)                     ||  |
|  ||                  *Ponto Crítico:* I/O Bound                   ||  |
|  +=================================================================+  |
|  ||                                                               ||  |
|  ||  +---------------------------------------------------------+  ||  |
|  ||  | 1. Deserialização (JSON/XML -> Objeto)                  |  ||  |
|  ||  |    *Ponto Crítico:* CPU Bound                           |  ||  |
|  ||  +---------------------------------------------------------+  ||  |
|  ||                                                               ||  |
|  ||  +---------------------------------------------------------+  ||  |
|  ||  | 2. Processamento, Validações e Regras de Negócio        |  ||  |
|  ||  |    *Ponto Crítico:* CPU Bound                           |  ||  |
|  ||  +---------------------------------------------------------+  ||  |
|  ||                                                               ||  |
|  ||  +---------------------------------------------------------+  ||  |
|  ||  | 3. Estruturação e Tracking do ORM                       |  ||  |
|  ||  |    *Ponto Crítico:* CPU Bound                           |  ||  |
|  ||  +---------------------------------------------------------+  ||  |
|  ||                                                               ||  |
|  ||  +---------------------------------------------------------+  ||  |
|  ||  | 4. Comunicação com o Banco (Persistência)               |  ||  |
|  ||  |    *Ponto Crítico:* I/O Bound                           |  ||  |
|  ||  +---------------------------------------------------------+  ||  |
|  ||                                                               ||  |
|  +=================================================================+  |
|  ||                  FECHAMENTO DE SESSÃO (ORM)                   ||  |
|  ||                  *Ponto Crítico:* I/O Bound                   ||  |
|  +=================================================================+  |
+-----------------------------------------------------------------------+
           |
           v
+-----------------------------------------------------------------------+
|                          BANCO DE DADOS SQL                           |
|                      *Ponto Crítico:* I/O Bound                       |
+-----------------------------------------------------------------------+
```

Percebemos ali que as operações I/O bound podem criar um "engarrafamento", se elas forem feitas de forma desleixada. No nosso futuro sistema estamos usando o código assíncrono justamente para burlar o máximo possível as limitações de I/O que existem.

Códigos assíncronos introduzem uma nova forma de lidar com I/O, pois distribuem de forma mais inteligente os recursos que não estão sendo usados, ou seja, introduzem operações não bloqueantes.
```
Tempo ----------------------------------------------------------------------------->

App / CPU  : [###] [###] [###] [###] [................] [###] [###] [###] [###]
             Req 1 Cons1 Req 2 Cons2  (Thread Livre)    Resp1 Ret 1 Resp2 Ret 2

Banco / IO :             [############################]
                                Busca Req 1
                                         [############################]
                                                  Busca Req 2
```

Agora temos consciência de conceitos sobre I/O assíncrono, mas ainda podemos melhorar a nossa forma de estruturar o nosso sistema. Pensando em uma aplicação que lida com uma altíssima escala de I/O, seria extremamente burro abrir uma sessão no banco para cada inserção a ser feita, pensando em uma alternativa mais inteligente, seria melhor inserir uma grande quantidade de elementos que foram acumulados durante um tempo `T` com apenas uma sessão aberta, fazer isso diminui o gargalo de sessões abertas e nos dá a opção de usar as sessões assíncronas, já que poderíamos abrir `M` sessões que teriam acumulado, todas elas, `N` elementos que seriam inseridos de forma instantânea -- isso escala muito bem, seria quase impossível exceder o connection-pool do SGDB dessa forma.
